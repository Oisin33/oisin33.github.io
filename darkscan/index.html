How it works

Dark patterns are deceptive interface choices that nudge you into decisions you wouldn’t otherwise make. Pure keyword matching catches the obvious ones but misses anything reworded, so DarkScan runs a cascade of detectors, cheapest first:

1 · Heuristics
regex / keyword rules, instant, precise on known phrasings
2 · Classifier
TF-IDF + logistic regression, generalises to unseen wording
3 · Embeddings*
semantic match, catches different vocabulary entirely
4 · LLM*
adjudicates the genuinely ambiguous cases

This demo runs tiers 1 and 2 (tiers 3-4 are server-side options in the full project, marked *). The classifier here is the actual trained scikit-learn model, its word + character n-gram vocabulary, IDF weights and regression coefficients are exported to model.json and the maths is re-implemented in detector.js, verified to match the Python model to within 0.0001.

Character n-grams are what let it generalise: “pay full price like a chump” shares sub-word features with the training example “pay full price”, so it’s flagged as confirmshaming even though that exact sentence was never in the data.

Under the hood, the full system

The browser demo above is one slice. The actual project is a full-stack application: a browser extension that scans live pages, a REST API that serves and persists results, a React analytics dashboard, and the Python detection engine with its learning loop.

Each tier only sees what the one above couldn't confidently settle. The cheap tiers handle the common cases, so most pages never reach the LLM at all, that's the cost-control argument for a cascade over a single big model. (*optional server-side tiers.)

System architecture

Browser extension
Manifest V3

One-click scan of the current tab. A content script extracts page HTML and posts it to the API; a popup renders findings in real time.

REST API
Flask · SQLite

Endpoints for scanning, history, stats, and the learning loop (/feedback, /distilled, /retrain). Connection-per-request SQLite.

Dashboard
React · Recharts

Scan history, severity trends, per-domain breakdowns, and a Review & Train console to confirm/reject detections and retrain.

Detection engine
Python · scikit-learn

Heuristic rules plus the tiered cascade. Segments a page into UI spans, classifies each, and merges results.

Design decisions

  • A cascade, not one big model. Keyword rules are precise but brittle; an LLM on every span is accurate but slow and costly. The cascade gets the LLM's quality on hard cases while paying for it only on hard cases.
  • TF-IDF before embeddings. Character n-grams already generalise to reworded patterns at near-zero cost and no model download, so they run before the heavier semantic tier, which only handles what they can't.
  • Per-span classification. Dark patterns are localised to one button or banner. Segmenting the page and classifying each element keeps the signal sharp instead of drowning it in whole-page text.
  • Browser-side inference for the demo. Exporting the model to run client-side means the public demo needs no server, can't run up a bill, and never sees user data.

Engineering challenges & bugs fixed

  • A real false-positive. The original "disguised ads" rule matched the substring brand, so Bootstrap's ubiquitous navbar-brand flagged a huge share of normal sites. Root cause: substring matching where token matching was needed. Narrowed to word-boundary ad tokens and added a regression test so it can't come back.
  • Feedback could poison the model. The feedback endpoint accepted any label string, so a typo like "Confirmshamming" would create a phantom class on the next retrain. Added server-side validation against the taxonomy.
  • Distillation risked model collapse. Letting the LLM's verdicts train the cheap tier is powerful but dangerous, training on a model's own guesses drifts toward its mistakes. Gated it: high-confidence only, separate store, excluded from training by default, human feedback always wins.

Testing

  • 19 backend tests, all runnable with no API key.
  • Cover the cascade routing (cheap tiers settle confident cases, uncertain ones escalate), the distillation guardrails (low-confidence verdicts aren't captured; human feedback overrides machine labels), and the detection logic including the navbar-brand regression.
  • The browser port has its own parity test asserting the JS classifier matches the Python model.

Reliability & safety

  • Confidence thresholds gate every escalation, so a tier only overrides a cheaper one when it's genuinely more certain.
  • Graceful degradation, if scikit-learn, the embedding model, or the LLM key is missing, that tier is skipped and the system still runs on whatever's available.
  • Distillation guardrails against model collapse, plus human-feedback validation against the taxonomy.

Deployment

  • Environment-based configuration, optional tiers toggle via env vars (DARKSCAN_USE_EMBEDDINGS, DARKSCAN_USE_LLM, DARKSCAN_DISTILL).
  • The public demo is fully static (the model exported to JSON), deployable to any static host or CDN with no backend.
  • The URL-scan feature ships a small serverless function (Cloudflare/Vercel) with a CORS-proxy fallback.

Scaling considerations , future work, not yet built

  • Versioned model releases so clients can pin a known model.
  • An incremental retraining pipeline driven by accumulated feedback, rather than full retrains.
  • Pushing more inference browser-side to keep server load flat as usage grows.

What I'd do differently

  • Store scans in a schema'd table from the start rather than JSON blobs, querying patterns across history would be cleaner.
  • Add evaluation telemetry early, so model changes could be measured against a held-out set automatically instead of by hand.
  • Build the feedback/review UI alongside the model, not after, the loop is only as good as how easily a human can correct it.

Honest limitations

  • The classifier is trained on a small seed dataset, it's a working proof of concept, not a production detector, and accuracy scales with more labelled data.
  • It analyses text. Purely visual dark patterns, a greyed-out decline button, misleading colour, are out of scope for the ML tier and left to the heuristics.
  • Per-category accuracy is uneven where the seed data is thinnest; the feedback loop is what closes that gap over time.
